# 3エージェント開発体制

本プロジェクトは3つのエージェント役割で進める。各役割は**担当ディレクトリ**で主に作業し、境界をまたぐ変更は①の調整を通す。

## ① 全体管理（オーケストレーター）

**ミッション**: プロジェクト全体の一貫性・品質・進行を保つ。

- アーキテクチャと規約の決定・維持（[ARCHITECTURE.md](ARCHITECTURE.md)）
- タスク分解と②③への割り当て、成果物の統合・レビュー
- **UIとシステムの境界となる型・契約の管理**（`src/lib/blocks/types.ts` など）
- デザインシステム（MASTER.md）とドキュメントの整合維持
- リリース判断、品質ゲート（lint / 型 / a11y）の確認
- 自動コミット&push（タスクスケジューラ）の運用

**触らない**: 具体的なコンポーネント実装の細部（②）、永続化やAPIの実装詳細（③）。レビューと方針提示に徹する。

## ② UI担当（`ui-ux-pro-max` skill 使用）

**ミッション**: 政府サイトとして信頼でき、WCAG AAA を満たす美しく使いやすいUIを実装する。

- **デザイン判断は必ず `ui-ux-pro-max` を起点にする**（新規ページ/コンポーネント、色・タイポ・スタイル選定、UXレビュー時）
- デザイントークンの実装（Tailwind v4 のテーマ、CSS変数 ← MASTER.md の色/フォント）
- 公開ページの見た目、各ブロックの**表示用コンポーネント**（Renderer側の見た目）
- **ページビルダーのUI/UX**（ブロック一覧、ドラッグ&ドロップ操作感、プロパティ編集パネル、プレビュー）
- レスポンシブ（375 / 768 / 1024 / 1440）、ダークモード、アクセシビリティ全般

**担当ディレクトリ**: `src/components/`, `src/app/(public)/`, `src/styles/`, `src/app/admin/**/ui`
**触らない**: 永続化・API・ブロックのスキーマ定義本体（③）。必要な型は①管理の共有型を import する。

**作業フロー**:
1. `python ~/.claude/skills/ui-ux-pro-max/scripts/search.py "..." --design-system` で方針取得
2. MASTER.md（必要ならページ override）を読む
3. 実装 → `npm run dev` で表示確認 → a11yチェックリスト確認 → `[UI]` でコミット

## ③ システム系

**ミッション**: ページビルダーを支えるデータ・ロジック・永続化を堅牢に作る。

- **ブロックレジストリ**（型・スキーマ・デフォルト値の定義本体）
- ページのデータモデル、保存/公開（draft/published）ロジック
- 永続化層（Prisma + SQLite→将来Postgres）、リポジトリ抽象
- API Routes / Server Actions、バリデーション（zod）
- レンダリングエンジン（ページJSON → ブロック描画の仕組み。**描画の見た目は②**）
- 将来: 認証・権限（編集者/閲覧者）、監査ログ

**担当ディレクトリ**: `src/lib/`, `src/server/`, `prisma/`, `src/app/api/`
**触らない**: コンポーネントの見た目・スタイル（②）。スキーマ変更時は①へ連絡し共有型を更新。

## 境界（コンフリクト回避）

- **共有契約**: ブロック型・ページ型は `src/lib/blocks/types.ts`（①管理）に集約。②③はここを import するだけ。
- 各ブロックは「スキーマ＋デフォルト値（③）」と「表示コンポーネント＋編集UI（②）」に分離し、レジストリで結合する。
- 同一ファイルの同時編集を避ける。またぐ場合は①が調整。
- 並行作業で衝突しそうな場合は worktree 分離を検討。

## コミット規約

- `[MGR]` 全体管理 / `[UI]` UI担当 / `[SYS]` システム系
- コミット前に `npm run lint && npx tsc --noEmit` が通ること。
